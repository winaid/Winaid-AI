## 1. 현재 상태 한 줄 요약

서비스는 작동하지만, **개인 소유 기반의 다중 플랫폼 구조**이기 때문에 회사 운영 체계로 정리가 필요한 상태다.
지금 바로 전면 이전하기보다, **사전 정리 후 단계적 이전**이 현실적이다.

| # | 리스크 | 설명 |
|---|--------|------|

| 5 | **다중 플랫폼 운영 복잡도** | 4개 이상의 외부 서비스에 분산 배포되어 있어 관리/장애 대응이 복잡함 (확인됨) |
| 6 | **결제 시스템 미완성** | DB 스키마와 일부 흔적은 있으나, 실제 결제 흐름이 동작하는지 추가 확인 필요 (추정) |

| 순서 | 할 일 | 이유 |
|------|-------|------|
| 1 | **플랫폼 계정 소유권 확인** | 회사 이관의 전제 조건. 누구 명의인지, 빌링은 어디로 가는지 파악 |
| 2 | **시크릿/API 키 감사** | 코드에 하드코딩된 키가 있는지, git 히스토리에 노출된 적 있는지 점검 |

| 4 | **외부/내부 기능 경계 파악** | 사용자가 보는 기능과 관리자 기능의 범위를 정리 |
| 5 | **AI 프록시 이중화 방안 검토** | 프록시 장애 시 전체 서비스 중단 방지를 위한 백업 방안 |

## 4. 하지 말아야 할 것

| # | 하지 말 것 | 이유 |
|---|-----------|------|
| 1 | **사용자향/관리자향 기능을 동시에 전면 이전** | 범위가 크고 리스크가 높음. 한쪽부터 순서대로 정리하는 것이 안전 |
| 2 | **시크릿 정리 없이 계정만 이동** | 키가 코드에 박혀 있으면 계정을 바꿔도 보안 문제가 해소되지 않음 |
| 3 | **검증 없이 이미지 생성 기능을 외부 확대** | 이미지 생성은 비용이 높고 프록시 안정성 의존도가 큼. 충분한 테스트 후 확대 |

---

## 5. 권장 방향

```
1단계  소유권 · 계정 · 시크릿 정리
       → 각 플랫폼 계정을 회사 명의로 확인/이전
       → 코드 내 시크릿 감사 및 환경변수 분리

2단계  사용자향(external/public) 기능 우선 정리
       → 랜딩페이지, 콘텐츠 생성, 결과 보기 등
       → 인증 체계 점검

3단계  관리자향(internal/admin) 기능은 후속 분리
       → AdminPage, 스타일 관리, 디버그 엔드포인트 등
       → 접근 통제 강화

4단계  프록시/이미지 검증 후 서비스 확대
       → AI 프록시 이중화 적용
       → 이미지 생성 안정성 확인 후 외부 노출 확대
```

## 6. 최종 판단

| 항목 | 판단 |
|------|------|
| 회사 이관 준비 상태 | **PARTIAL** — 기술적으로 동작하나 소유권/시크릿/경계 정리가 선행되어야 함 |
| 사용자향(external) 이전 준비 | **PARTIAL** — 기능은 존재하나 인증 경계와 시크릿 정리 후 가능 |
| 관리자향(internal) 동시 이전 권장 여부 | **NO** — external 정리 후 별도 진행이 안전 |

**가장 중요한 한 줄:**
지금 필요한 것은 "개발"이 아니라 **"정리"**다. 소유권과 시크릿을 먼저 정리하면, 이후 이전은 단계적으로 진행할 수 있다.
