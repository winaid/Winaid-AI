### 2.2 `34363ac` — PSI 타임아웃 정합 (25s → 70s)

**파일**: `next-app/app/api/diagnostic/route.ts` (+4/−1)

**문제**: AEO/GEO 진단의 PSI(PageSpeed Insights) 호출이 느린 사이트(Lighthouse Mobile 25-40초)에서 항상 실패. `route.ts`가 `withTimeout(fetchPsi, 25_000)`로 감싸는데 내부 `psi.ts`는 35초×2회 시도 = 최악 70초 budget로 설계됨. 외부 wrapper가 먼저 abort → 재시도 코드는 dead retry.

**변경**: wrapper budget을 `25_000` → `70_000`으로 변경. `psi.ts` 코멘트의 설계 의도와 정합.

**예산 검증**: crawl 40s + psi 70s + enrich 40s = 150s. Vercel `maxDuration=180` 안에 정합.

**롤백**: revert commit `34363ac`.

**부수 개선 후보** (이번엔 미적용):
- `withTimeout`에 `AbortController` 전파해 wrapper abort 시 inner fetch도 cancel — 현재는 백그라운드에서 inner promise가 계속 돌아 리소스 낭비
- PSI를 crawl과 부분 병렬화

### 2.6 `89a8e9f` — 비임상 행동 시 임상 segment deterministic strip (옵션 A)

**파일**: `next-app/app/api/image/route.ts` (+59/−4)

**문제**: 2.5 (HARD OVERRIDE) 적용 후에도 일부 케이스에서 진료의자 이미지 재발. 추적 결과 진짜 원인은 **blog-core/buildImagePrompt가 임상 묘사를 baked in해서 보냄**:
1. `categoryHints['치과'] = 'dental clinic setting, modern minimalist Korean dental office'` — prefix
2. `SCENE_VARIANTS` 6개 모두 임상 (`'examination chair'`, `'close-up of dental tools and procedure'`, `'wide view of bright modern clinic interior'` 등). `sectionIndex >= 2`면 alt 옆에 그대로 append
3. `'Korean patient context'` suffix — 환자 framing 강제

이 **구체 묘사**가 모델에서 abstract HARD OVERRIDE보다 더 강하게 받아들여져 우회.

**변경**: `body.prompt`를 모델에 보내기 전 서버에서 deterministic 처리:
1. **비임상 행동 키워드 감지** (한·영 정규식 11개): 양치/치솔질, 치실, 치간 칫솔, 가글/구강세정, 식사/먹/섭취, 음용/마시, 산책, 운동/스트레칭/재활, 수면/잠, 복용/약, 스킨케어/세안
2. 매치 시 `body.prompt`를 쉼표 split → **임상 keyword 포함 segment 제거**
   - keyword pattern: `clinic | dental {office|tools|scan|procedure|chair} | examination {chair|area|environment|setting|room} | operatory | consultation {environment|setting|room} | treatment {environment|chair|setting|room} | reception desk | X-ray imaging | dentist {explaining|focused} | patient {consultation|receiving treatment|context} | medical clinic interior`
3. alt 텍스트 segment 와 중립 segment(lighting·atmosphere)는 보존
4. 모든 segment가 임상으로 분류되어 빈 문자열이 되면 원본 유지 (false-positive 가드)
5. 처리 사실은 `[BLOG_IMAGE] 비임상 행동 감지 → 임상 segment strip (N→M chars)` 로그로 가시화

**예시**:
- Before: `dental clinic setting, modern minimalist Korean dental office, 회복기 죽 식사, wide view of bright modern clinic interior, examination chair, soft natural light, Korean patient context, warm approachable atmosphere`
- After: `회복기 죽 식사, soft natural light, warm approachable atmosphere`

**롤백**: revert commit `89a8e9f`. 위치는 `route.ts` POST handler 위에 module-level helpers + isBlogMode 가드 안에서만 적용.

**모니터링**: Vercel 로그에서 `[BLOG_IMAGE] 비임상 행동 감지` 라인 빈도 추적. 잦으면 옵션 A가 살리고 있다는 증거. 블로그 본문에서 임상 토픽 글의 임상 segment가 잘못 strip되는 false-positive 케이스 (코드는 isNonClinicalAction이 참일 때만 실행되므로 이론상 없음, 실측 확인 필요).

**옵션 C(blog-core 직접 정리)와의 관계**: 시니어가 buildImagePrompt 자체를 정리하면(`categoryHints` location-free화 + SCENE_VARIANTS action-aware화) 본 strip은 의미가 줄어듦. 그때 본 helper 제거 가능.

## 5. 시니어 결정 대기 (Decision Queue)

직접 main 머지 외에도 다음 사항은 시니어 판단이 필요합니다:

1. **머지 흐름** — PR + CI 통과 의무화로 돌릴지, 아니면 본 세션처럼 직접 fast-merge 유지할지. README:241에서 GitHub Actions가 `tsc --noEmit + next build`를 PR 시 돌린다고 명시 → 직접 머지는 그 가드를 우회

3. **monorepo 방향** — `ARCHITECTURE_PLAN.md`가 monorepo 계획 폐기를 명시. 그대로 두 앱 fork로 갈지, `packages/` 추출 다시 시도할지

5. **지속 모니터링 채널** — 본 세션이 추가한 텔레메트리 console 로그를 어떤 도구(Sentry / Datadog / 자체)로 수집할지
